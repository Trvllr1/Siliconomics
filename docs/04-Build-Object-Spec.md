# Siliconomics Build Object Specification

## Core Domain Model

### Version 1.1 — Amended 2026-07-17

---

# Purpose

The Build is the central object of Siliconomics.

Every meaningful activity within the platform either creates, analyzes, compares, reviews, exports, or archives a Build.

Siliconomics does not organize work around spreadsheets, presentations, projects, or reports.

It organizes work around immutable Builds.

The Build is the platform's shared engineering and commercial object.

---

# Core Philosophy

A Build represents one complete engineering scenario.

It captures engineering intent, deterministic computation, and resulting commercial outcomes at a specific point in time.

A Build is not a file.

It is not merely a collection of calculations.

It is a complete engineering decision object.

Every downstream artifact derives from a Build.

---

# First Principles

## Principle 1 — Identity

Every Build possesses a permanent identity.

Each Build shall contain, at minimum:

* Build ID (UUID)
* Build Name
* Description
* Creator
* Organization
* Portfolio
* Creation Timestamp
* Parent Build (optional)
* Build Version
* Status

The Build ID never changes.

---

## Principle 2 — Immutability

Once generated, a Build shall never be modified.

Any engineering change, regardless of size, produces a new Build.

Examples include:

* Die size changes
* Process node changes
* Packaging changes
* Yield assumptions
* Financial assumptions
* Program assumptions
* Manufacturing assumptions

History is preserved through lineage rather than mutation.

---

## Principle 3 — Determinism

A Build represents deterministic engineering computation.

Given identical:

* Design Model
* Formula Library
* Reference Models
* Engine Version

the Build shall always produce identical outputs.

The Build never depends upon randomness.

---

## Principle 4 — Reproducibility

Years after creation, a Build must be reproducible.

Siliconomics shall preserve:

* Formula Library version
* Reference Model version
* Calculation engine version
* User inputs
* Build metadata

Re-running a historical Build under identical conditions must produce identical results.

---

## Principle 5 — Traceability

Every result within a Build shall explain its origin.

Users must always be able to determine:

* which inputs produced the result
* which formulas were applied
* which assumptions were used
* which reference models participated
* which engine version executed the calculations

Nothing within a Build should be opaque.

---

# Build Lifecycle

Every Build follows the same lifecycle.

```text
Design Model

↓

Validation

↓

Build Generation

↓

Deterministic Computation

↓

Snapshot Creation

↓

Review

↓

Comparison

↓

Approval

↓

Export

↓

Archive
```

A Build never returns to the Design Model.

Subsequent work always creates a new Build.

---

# Build Composition

Every Build contains six logical sections.

## 1. Intent

Captures user intent.

Examples:

* Process node
* Die area
* Packaging
* Yield assumptions
* Financial assumptions
* Program assumptions
* Technology selections

Intent records what the engineer requested.

**Shipped composition members added since v1.0:**

- `architecture` — ArchitectureComposition with block-level BOM (NRE, licenses, royalties, schedule impact, verification effort, supply chain risk).
- `dataVintage` — DataVintage tracking the reference model version, packaging model version, and commodity price date used at computation time. Ensures reproducibility by anchoring the reference data snapshot.
- `timeModel` — Optional TimeModel enabling time-dimension projection. Contains D0 yield-learning curve parameters, ramp shape/duration, annual ASP erosion rate, max quarterly supply, volume allocation profile, projection horizon, and respin risk configuration. When present, `computeBuildMetrics` includes time-phased metrics (break-even quarter, program constraint, respin-adjusted expected net). When absent, the engine produces the exact static lifetime values (equivalence theorem).

---

## 2. References

Captures every external dependency.

Including:

* Formula Library version
* Reference Model version
* Cost database version
* Manufacturing assumptions
* Packaging assumptions

References guarantee reproducibility.

---

## 3. Computation

Stores deterministic outputs generated by the computation engine.

Examples:

* Dies per Wafer
* Yield
* Bare Die Cost
* Package Cost
* NRE
* Unit Cost
* Break-even
* ROI
* Schedule estimates

Computation results are immutable.

---

## 4. Evidence

Documents how results were obtained.

Includes:

* Formula references
* Intermediate calculations
* Dependency graph
* Calculation provenance

Evidence enables explainability.

---

## 5. Metadata

Administrative information.

Examples:

* Creator
* Organization
* Portfolio
* Permissions
* Tags
* Build status
* Revision lineage

---

## 6. Deliverables

Everything produced from the Build.

Examples:

* Engineering Report
* Executive Report
* Cost Summary
* Manufacturing Summary
* CSV Export
* PDF Export
* Presentation Package

Deliverables inherit the Build ID.

---

# Parent–Child Relationships

Builds may derive from earlier Builds.

Example:

```text
Build 101

↓

Increase SRAM

↓

Build 102

↓

Switch to 3nm

↓

Build 103

↓

Move to CoWoS

↓

Build 104
```

Each Build preserves its parent relationship.

Historical evolution is never lost.

---

# Snapshot Relationship

Every Build owns exactly one Snapshot.

The Snapshot represents the computed state of that Build.

Snapshots are immutable.

Snapshots never exist independently of Builds.

---

# Comparison Relationship

Comparisons evaluate Builds.

Siliconomics never compares spreadsheets.

Siliconomics compares Build objects.

Comparisons may evaluate:

* Area
* Yield
* Cost
* Packaging
* Schedule
* Risk
* ROI
* Break-even
* Manufacturing assumptions

---

# Portfolio Relationship

Portfolios organize Builds.

Portfolios do not contain engineering state.

Engineering truth always resides within individual Builds.

---

# Collaboration

Collaboration occurs around Builds.

Users may:

* comment on Builds
* review Builds
* approve Builds
* reject Builds
* share Builds
* present Builds

Collaboration never alters Build contents.

---

# Explainability

Every metric within a Build shall provide:

* definition
* governing formula
* contributing inputs
* calculation pathway
* supporting assumptions

Siliconomics shall never present unexplained engineering or financial results.

---

# Export Behavior

Every exported artifact shall reference:

* Build ID
* Formula Library version
* Reference Model version
* Creation timestamp
* Platform version

Exports remain traceable to their originating Build.

---

# AI Interaction

Future AI systems do not modify Builds.

They may:

* analyze Builds
* compare Builds
* summarize Builds
* forecast from Builds
* recommend alternative Builds

All AI functionality remains downstream of deterministic computation.

---

# Architectural Rule

No platform capability may bypass the Build.

All workflows—including reporting, comparison, collaboration, forecasting, optimization, and future AI services—must originate from a valid Build object.

The Build is Siliconomics' single engineering and commercial reference.

---

# Success Criteria

A Build succeeds when it enables every stakeholder—engineering, manufacturing, finance, program management, procurement, and executive leadership—to reason from the same deterministic engineering object.

The objective is not merely to calculate outcomes.

The objective is to establish shared engineering truth from which better decisions can be made.

---

# Guiding Principle

> **Every Design Model becomes a Build. Every Build becomes shared engineering truth. Every shared truth enables better decisions.**

This principle governs every future evolution of Siliconomics.
